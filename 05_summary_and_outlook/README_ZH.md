# 05 总结与展望（Summary and Outlook）

本目录是整个项目的**解释与收口层**，用于：

- 对实验结果进行高层总结
- 解释 prompt drift 的行为模式
- 讨论方法层面的启示与局限
- 明确哪些结论**不能**从本实验中推出
- 给出未来可扩展方向

> 本目录**不引入新的实验或指标**。所有结论必须可追溯到 `04_results/` 的结果工件，以及 `03_evaluation_rules/` 的评测协议。

---

## 1) 分工边界（该放什么 / 不该放什么）

### 属于 `05_summary_and_outlook/` 的内容

- **结果层总结**（模式、趋势、定性观察）
- **行为解释**（基于结果，而非主观猜测）
- **方法论启示**（这个评测框架擅长/不擅长什么）
- **非结论声明**（明确不能声称的点）
- **未来工作方向**（扩展，而非承诺）

### 不属于本目录的内容

- 原始模型输出 → `04_results/01_raw_model_outputs/`
- 评分 JSON / CSV 表 → `04_results/02_cross_model_evaluation/`
- 评测规则与 judge 逻辑 → `03_evaluation_rules/`
- 提示词文本或变体 → `02_prompt_variants/`

---

## 2) 推荐写作结构

建议任何总结类文档遵循以下顺序：

1. **实验范围说明**  
   本总结覆盖哪些实验与结果子集。

2. **关键观察**  
   在不同模型 / prompt / judge 下重复出现的现象。

3. **失败模式与敏感性**  
   指令遵循在哪些情况下不稳定。

4. **方法论层启示**  
   对 prompt 稳定性评测方法本身的反思。

5. **局限与非结论**  
   防止结果被过度外推。

6. **展望 / 扩展方向**  
   未来可加入的变量、协议或分析方式。

---

## 3) 证据纪律（非常重要）

- 每一条结论都应**可追溯**到：
  - `04_results/` 中的表或 JSON，或
  - `03_evaluation_rules/` 中明确的评测协议。

- 推荐使用表述：
  - “我们观察到……”
  - “在该评测协议下，结果显示……”

- 避免能力断言、因果推断，除非有明确证据支撑。
